code-quality assessment

A phased plan to replace YURA's primitive quality heuristics with a multi-dimensional, statically-grounded, calibrated assessment — every change anchored in the 2024–2026 arXiv corpus (165 papers, indexed).

The problem we're fixing

Today YURA judges quality two ways, both fragile. The corpus identifies exactly why each fails.

Regex "shitty commit" heuristic

Keyword match on commit messages + a one-line-touch rule. Catches lazy messages, says nothing about the code.

analysis.py · _is_shitty()

One-shot LLM judge, vague rubric

Single pass scores 3 axes from a one-sentence prompt. AXIOM: this hallucinates flaws, can't tell a tweak from a refactor, and absolute-score agreement is low (MCC<15).

The plan — five phases

Sequenced by leverage. Phases 1–2 attack the hallucination problem head-on and need no new infrastructure. Each phase ships independently and improves the report on its own.

01

RACE rubric — explicit, anchored dimensions

Replace the one-sentence rubric with gradeable per-axis criteria.
Effort: SROI: Highest
  • Score four established axes — Readability, mAintainability, Correctness, Efficiency — plus authenticity, each with concrete anchors ("0–20: no error handling at boundaries; 80–100: …").
  • Prompt-only change in llm.py; widen the JSON schema & _normalize(); surface the axes in the report exhibits.
  • Keep scores pointwise with a reference anchor — the reliability papers show this minimizes verbosity/position bias.
Grounding: RACE benchmark (2508.14419, 2603.25146), ISO 25010; anchoring & pointwise from 2606.01629, 2605.28710.
02

Static-analysis grounding

Feed objective tool findings into the judge as evidence — stop guessing from the diff alone.
Effort: MROI: High
  • Run pylint + bandit (Python) / eslint (JS) on changed files at the touched revision; map findings → RACE axes (Pylint convention→readability, error→correctness, refactor→maintainability; Bandit→security).
  • Inject the findings into the judge prompt so scores are anchored to real issues, not hallucinated ones.
  • New services/static_analysis.py; language-detection by extension; graceful skip for unsupported languages.
Grounding: "Static Analysis as a Feedback Loop" (2508.14419) — deterministic tool→dimension mapping; BitsAI-CR (2501.15134).
03

Git-native risk signals

Turn data we already parse into a per-commit defect-risk score.
Effort: MROI: High · YURA-unique
  • Compute Just-In-Time risk features per commit: relative churn, files touched, diff dispersion, message↔diff alignment, hotspot recency.
  • Combine into a Diff Risk Score shown per contributor and used to weight the watch-list — a cheap, non-LLM signal that works even in free git-stats mode.
  • Extends analysis.py; no API cost.
Grounding: churn→defect density (Nagappan & Ball); DeepJIT/CC2Vec, JITLine via 2605.17548; Meta Diff Risk Score (2605.30208).
04

Real AI-code detection

Replace the keyword regex with research-grade signals.
Effort: M–LChoice needed
  • Tier 1 (cheap, ship first): feature-based stylometry — whitespace ratios, identifier conventions, structural-complexity regularity. No model, runs locally.
  • Tier 2 (optional): embedding classifier (GraphCodeBERT, LLMSniffer-style) or perplexity/burstiness — higher accuracy, adds a dependency/model.
Grounding: "Whitespaces Don't Lie" (2601.19264), LLMSniffer (2604.16058), CodeVision (2501.03288), DetectGPT-style perplexity.
05

Judge reliability & cost control

Make scores trustworthy and cheap at scale.
Effort: MCost-sensitive
  • Two-pass triage: deterministic rules resolve clear-cut commits (~70%); the LLM only adjudicates the ambiguous ~30% — large cost reduction (BitsAI-CR).
  • Bias mitigation: blind the author, control for length, optionally repeat/aggregate; favor precision over recall (false positives erode trust).
  • Optional panel: vendor-diverse multi-judge with consensus + a reliability stat for high-stakes runs.
Grounding: two-pass & precision (2501.15134, 2412.18531); bias (2606.22329, 2605.25273); panels (2605.20351).

Decisions I need from you

1 · Scope of first cut

Recommend shipping Phases 1 + 2 together — the rubric rewrite plus static grounding is the combination AXIOM says fixes hallucinated scores, and it needs only pylint/bandit subprocess calls.

Options: (a) 1+2 together [recommended] · (b) 1 only, evaluate, then 2 · (c) different order

2 · Language coverage for static analysis

Pylint/Bandit are Python-only. Your test repos (chainlit, click) are Python, but real targets are mixed.

Options: (a) Python first, add eslint/others later [recommended] · (b) multi-language from day one · (c) language-agnostic LLM-only grounding

3 · AI-detection depth (Phase 4)

Tier-1 stylometry is free and local. Tier-2 (GraphCodeBERT embeddings) is materially more accurate but adds a model dependency + load time.

Options: (a) Tier-1 only for now [recommended] · (b) Tier-1 now, Tier-2 behind a flag · (c) go straight to Tier-2

4 · Multi-judge panel (Phase 5)

A vendor-diverse panel improves reliability but multiplies cost per commit. Worth it only for high-stakes reviews.

Options: (a) single judge + two-pass triage [recommended] · (b) optional panel toggle in Settings · (c) skip panels entirely
